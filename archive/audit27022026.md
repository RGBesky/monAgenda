# PARTIE 1 : AUDIT DE L'EXISTANT (ÉTAT DES LIEUX)

Basé sur l'analyse du code source (structure, dépendances, logique métier et sécurité).

## 1. 🛡️ Sécurité (Priorité : Haute)

**État actuel : Globalement bon sur le stockage local, mais faille critique sur l'export.**

*   ✅ **Points Forts :**
    *   **Stockage des identifiants :** Utilisation correcte de `flutter_secure_storage` avec l'option `encryptedSharedPreferences: true` sur Android.
    *   **SSL/TLS :** `Dio` utilisé sans désactivation de la vérification SSL (pas de `badCertificateCallback`).
    *   **Isolation :** Clés API non hardcodées, configurables via l'UI.

*   🚨 **Faille Critique (A corriger immédiatement) :**
    *   **Export de configuration non sécurisé :** Dans `lib/providers/settings_provider.dart`, la méthode `toExportJson()` inclut `infomaniakAppPassword` et `notionApiKey` en clair (encodage Base64 simple).
    *   **Risque :** Compromission totale des comptes si le QR code est intercepté.
    *   **Correction requise :** Chiffrement AES-256 avec mot de passe utilisateur avant l'export.

## 2. 🎨 UX (Expérience Utilisateur)

**État actuel : Moderne et réactif, orienté "Power User".**

*   ✅ **Points Forts :** Thématisation `flex_color_scheme`, Architecture `Riverpod`, Support Desktop réel (`sqflite_common_ffi`).
*   ⚠️ **Points d'amélioration :** Écrans d'erreur techniques ("Erreur: $e"), Widget Android basique, Internationalisation rigide.

## 3. 🖱️ Ergonomie & Architecture

**État actuel : Architecture propre et maintenable.**

*   ✅ **Points Forts :** Séparation `core/features`, Modèle de données solide (contraintes UNIQUE).
*   ⚠️ **Points d'amélioration :** Gestion des participants stockée en JSON texte (lent à parser pour la recherche), pas de table de jointure.

# PARTIE 3 : INSTRUCTIONS D'EXÉCUTION POUR OPUS 4.6

> **À l'attention de l'IA Opus :** Voici ton plan de bataille séquentiel. Ne passe pas à la phase suivante tant que la précédente n'est pas validée.

## 🔵 Phase 1 : Correctifs Critiques (Sécurité & Fondations)

1.  **Sécuriser l'Export (AES-256)**
    *   Modifier `SettingsProvider` pour utiliser le package `encrypt`.
    *   Implémenter un flux où l'utilisateur saisit un mot de passe temporaire pour générer le QR Code.
    *   Le QR Code généré ne doit contenir **aucune** donnée lisible en clair.

2.  **Créer le Scanner d'Import Mobile**
    *   Ajouter `mobile_scanner`.
    *   Créer une vue `ImportConfigScreen` qui scanne, demande le mot de passe, déchiffre le JSON et applique les settings.

3.  **Réparer les Notifications (WorkManager)**
    *   Implémenter `workmanager` pour Android.
    *   S'assurer que les notifications programmées survivent au redémarrage du téléphone et au "kill" de l'application.

4.  **Hardcoding Locale**
    *   Dans `main.dart`, forcer `Intl.defaultLocale = 'fr_FR'` et initialiser les timezones sur `Europe/Paris` (supprimer la logique de détection auto si fragile).

## 🟡 Phase 2 : Architecture V2 (Offline-First)

1.  **Implémenter la "Sync Queue"**
    *   Créer une table SQL `sync_queue` (id, action, payload, timestamp).
    *   Modifier les Repository pour :
        *   Ecrire dans la base locale `events` (Optimistic UI : l'user voit l'event tout de suite).
        *   Ecrire l'action dans `sync_queue`.
    *   Créer un `SyncWorker` qui dépile la queue quand `Connectivity` est OK.

2.  **Gestion des Erreurs Silencieuses**
    *   Ajouter une table `system_logs`.
    *   Ajouter un indicateur visuel (ex: icône attention rouge) dans les Paramètres si des erreurs de sync existent.

3.  **Validation du Schéma Notion**
    *   Ajouter un check au démarrage : vérifier que les colonnes `Date` (ou celle configurée), `Tags` et `Status` existent bien dans la Database Notion cible.

## 🟢 Phase 3 : IA Locale & Futur (Vision V3)

1.  **Intégration Moteur IA**
    *   Configurer `tflite_flutter`.
    *   Créer un service de téléchargement pour récupérer le modèle **Danube 3** depuis une URL sécurisée (kDrive public ou GitHub Releases).
2.  **Service NLP**
    *   Implémenter le parsing du texte brut vers `EventModel` en utilisant le modèle local.
