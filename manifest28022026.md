### Décision 1 : Suppression définitive des plugins sociaux natifs
**Décision :** Retrait de `flutter_contacts` et `share_plus`. Exécuté et commité.

**Argumentaire :**

**Impact codebase :** `pubspec.yaml` ✅, `AndroidManifest.xml` ✅, `platform_utils.dart` ✅ (flag `supportsContacts` retiré). Commite : `choix archi: drop flutter_contacts and share_plus...`.

### Décision 2 : Architecture IA 100% locale (llama.cpp + GGUF 4-bit + GBNF)

**Décision :** L'inférence NLP ("Saisie Magique") sera 100% on-device, via `llama.cpp` en FFI Dart. Aucune API Cloud. Aucun TensorFlow Lite.

**Argumentaire :**
- TFLite (`tflite_flutter`) n'est pas conçu pour les LLM génératifs (architectures transformer auto-régressives). Format GGUF incompatible.
- Les API Cloud (OpenAI, etc.) violent la promesse de souveraineté et de confidentialité de l'application.
- `llama.cpp` (C++) est le moteur d'inférence de référence pour les SLM on-device : optimisé CPU/GPU mobile, supporte GGUF, supporte la contrainte GBNF strict.

**Modèle retenu (décision finale) : H2O Danube 3 — 500M (q4_k_m)**
Justification du choix vs Qwen2.5-0.5B :
1. Fine-tuné spécifiquement sur extraction d'information structurée — use case exact de la Saisie Magique.
2. Meilleure conformité GBNF à ce niveau de quantification : moins de "token fuite" hors grammaire.
3. Gestion du français court sans drift vers l'anglais — critique pour le system prompt francophone.

**Paramètres du modèle retenus :**
- Famille : **H2O Danube 3 (500M)** — Qwen2.5-0.5B explicitement écarté
- Format : **GGUF quantisé 4-bit (q4_k_m)**
- Poids : **~400 Mo maximum**
- Déploiement : **Téléchargement on-demand** au premier clic "Magie" (pas embarqué dans l'APK/DEB)
- Intégrité : hachage **SHA-256** à vérifier après téléchargement (prévention MitM)

**Contrainte d'exécution absolue :** Tout appel à `llama.cpp` doit s'effectuer dans un `Isolate.run(...)` Dart. L'Event Loop Flutter (60FPS) ne doit jamais être bloqué par l'inférence.

**Contrainte de sortie absolue :** Utiliser une grammaire **GBNF** passée au contexte `llama.cpp` pour forcer la génération exclusive de JSON valide correspondant à `EventModel`. Aucun token de verbiage, aucune hallucination textuelle possible.

### Décision 4 : QR Code — Pipeline exact (correction erreur V3GK section 7)

**Pipeline correct (décision + code existant) :**

**Ce qui est périmé (à ne JAMAIS utiliser) :**

### Décision 6 : Correction des bugs et incohérences du code existant

**Bugs identifiés lors de l'audit V2 (28/02/2026), non encore corrigés :**

**Bug 1 — Déduplication `sync_queue` (consommation API gâchée)**
Si un event est modifié 3 fois consécutivement en mode offline, la `sync_queue` stocke 3 lignes `action=UPDATE` au lieu d'écraser la précédente. Au retour réseau, 3 requêtes API partent au lieu d'une.
*Correction :* Dans `DatabaseHelper.enqueueSync()`, avant d'insérer une nouvelle ligne `UPDATE`, exécuter un `DELETE FROM sync_queue WHERE event_id = ? AND action = 'UPDATE' AND status = 'pending'` pour écraser les mises à jour antérieures non encore envoyées.

**Bug 2 — `deleteEvent` scope pagination (désync fantôme)**
La méthode `deleteEvent` cherche l'event dans la "plage affichée" en mémoire. Si l'event a été scrollé hors vue avant d'être supprimé, il est absent du state local mais toujours en base SQLite → il ne rejoint jamais la `sync_queue` → il persiste côté CalDAV/Notion.
*Correction :* `deleteEvent(id)` doit toujours travailler par `id` en base SQLite directement, sans dépendre du state en mémoire. La suppression SQLite (`is_deleted = 1`) et l'enqueue doivent se faire via `DatabaseHelper` indépendamment de la vue courante.

**Bug 3 — Recherche plein texte sur JSON brut (lenteur croissante)**
Les colonnes `participants` et `description` de la table `events` stockent du JSON sérialisé (`jsonEncode`). Une recherche par mot-clé (`LIKE '%mot%'`) sur du JSON brut contient les artefacts de sérialisation et ralentit proportionnellement à la croissance de la base.
*Correction à évaluer :* Activer SQLite FTS5 (`CREATE VIRTUAL TABLE events_fts USING fts5(title, description, participants, content=events, content_rowid=id)`) et maintenir la table synchronisée via des triggers `AFTER INSERT/UPDATE/DELETE`.

**Bug 4 — RAM modèle IA : déchargement après inférence**
Un modèle GGUF 4-bit chargé en mémoire via `llama.cpp` occupe ~1 à 1.2 Go de RAM. Sur un appareil Android avec 4-6 Go, l'OS Android (LMK) tue l'application en arrière-plan sans prévenir.
*Correction :* Le `MagicEntryService` doit implémenter un cycle `load → infer → unload` : charger le contexte juste avant l'inférence, exécuter dans l'`Isolate`, libérer explicitement le contexte (`llama_free_model()` + `llama_free()`) immédiatement après la réponse. Ne jamais garder le contexte IA en mémoire entre deux appels.

**Bug 5 — kDrive backup endpoint en erreur**
L'endpoint de dépôt du backup vers kDrive est signalé non trouvé dans l'audit V2. Le backup souverain est une promesse clé de l'application.
*Action :* Vérifier les credentials FTP/WebDAV kDrive dans `InfomaniakService`, valider l'URL du endpoint, ajouter un test de connexion dans l'écran de réglages (bouton "Tester la connexion kDrive") avec retour visuel clair.

### Décision 7 : SQLite chiffré au repos (sqflite_sqlcipher)

**Décision :** Remplacer `sqflite` par `sqflite_sqlcipher`. La base de données locale est chiffrée AES-256 at rest. Niveau de sécurité équivalent à un gestionnaire de mots de passe.

**Argumentaire :**
- La base SQLite contient des données potentiellement sensibles : titres de réunions, noms de participants, clés de session sync_queue.
- Sur un appareil Android déverrouillé (ou Linux desktop), un simple `adb pull` ou accès fichier suffit à lire la base `sqflite` non chiffrée.
- `sqflite_sqlcipher` est un drop-in replacement : même API que `sqflite`, ajoute uniquement le paramètre `password`.
- La clé de chiffrement est dérivée via PBKDF2 depuis un secret stocké dans `flutter_secure_storage` clé `db_encryption_key`.

**Implémentation :**
```dart
// Dans DatabaseHelper.open() :
final dbKey = await _secureStorage.read(key: 'db_encryption_key') ?? _generateAndStoreKey();
final db = await openDatabase(path, password: dbKey, version: kCurrentDbVersion, ...);
```
- La clé est générée aléatoirement au premier lancement (32 octets base64) et stockée dans `flutter_secure_storage`.
- Migration depuis une base non chiffrée (utilisateurs existants) : détecter si la base est lisible sans mot de passe → exporter, recréer chiffrée, réimporter.

**Impact :** `pubspec.yaml` : remplacer `sqflite` par `sqflite_sqlcipher`. Aucun autre changement d'API.

## PARTIE 5 — RÈGLES ABSOLUES POUR OPUS 4.6 (NE JAMAIS ENFREINDRE)

Ces règles s'appliquent à **tous les prompts** et à toute contribution au projet `monAgenda` :

1. **State Management :** Exclusivement `flutter_riverpod`. Jamais `provider`. Utiliser `AsyncNotifier` pour tout state asynchrone.

2. **Accès credentials :** Toujours via `ref.read(settingsProvider)`. Jamais d'instanciation directe de `FlutterSecureStorage` ailleurs que dans `SettingsNotifier`.

3. **Réseau :** L'état offline est lu via `ref.watch(isOfflineProvider)`. Ne pas dupliquer la logique de `connectivity_plus`.

4. **Thread safety IA :** Tout appel `llama.cpp` dans un `Isolate.run(...)`. Jamais sur le thread principal.

5. **Plugins supprimés :** Ne JAMAIS réinstaller `flutter_contacts` ou `share_plus`. Ne JAMAIS ajouter `READ_CONTACTS` dans `AndroidManifest.xml`.

6. **QR Code :** Le pipeline est `AES-256 via CryptoUtils.encryptToExportString`. Ne pas recréer de méthode `gzip+base64` (c'est `@Deprecated`).

7. **Nomenclature CalDAV :** La classe s'appelle `InfomaniakService`, pas `CalDAVClient`. La table s'appelle `events`, pas `events_cache`.

8. **Desktop vs Mobile :** Les features Time Blocking, Meeting Note, Export PPT, Smart Context Parser sont **Desktop ONLY**. Utiliser `LayoutBuilder` ou `Platform.isLinux/isWindows` comme garde-fou.

9. **Isolation des couches :** Les `Notifier` Riverpod ne font pas d'appels UI directs. Les Widgets ne lisent pas SQLite directement.

10. **Logging :** `print()` est interdit en production. Utiliser `AppLogger` (qui écrit dans la table `system_logs`).

11. **SQLite chiffré :** La base de données locale utilise `sqflite_sqlcipher`. Ne jamais utiliser `sqflite` non chiffré. La clé de chiffrement est toujours lue depuis `flutter_secure_storage` clé `db_encryption_key`. Ne jamais passer `password: null` à `openDatabase()`.
